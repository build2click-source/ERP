Product Requirements Document (PRD): Finance, Inventory, Invoicing & Clients Module
Project Name: Enterprise ERP System
Design Philosophy: The Monolith of Precision
Target Region: India (State Context: West Bengal / Bagda)
Document Version: 2.0 (Final)
1. Executive Summary
Goal: Deliver an integrated ERP module that provides an immutable double‑entry ledger, product catalog, inventory costing (FIFO/LIFO), GST‑aware invoicing with e‑invoicing support, and a comprehensive client management/onboarding system. The system must guarantee accounting integrity (Assets = Liabilities + Equity), full auditability, and operational performance via month‑end snapshots and targeted indexes.
Primary Users: Finance team, Sales, Warehouse operations, Onboarding agents, and Auditors.
Key Non-Functional Requirements: * ACID transactional integrity.
Append‑only financial records with tamper evidence.
Strict Role-Based Access Control (RBAC).
Data privacy compliance for KYC documents.
High availability for high-volume posting operations.
2. Objectives and Scope
Accounting: Implement an immutable double‑entry ledger with database‑enforced zero‑sum per transaction and contra-entry correction flows.
Invoicing: Create tax‑aware invoices that auto‑generate balanced ledger transactions and support IRN/QR generation via GSP/IRP.
Products & Inventory: Maintain a robust product catalog, dynamic cost layers, and FIFO/LIFO consumption with automatic COGS (Cost of Goods Sold) posting.
Clients: Manage the full client lifecycle (create, edit, onboard, KYC, link to AR/AP accounts) with a comprehensive audit history.
Performance: Utilize month‑end snapshots and targeted indexing to keep balance queries highly performant.
Compliance: Ensure strict GST mapping, e‑invoicing compliance, e‑way bill data capture, and an immutable audit trail.
3. UX & Design System: "The Monolith of Precision"
The interface must feel organized, dense, and unshakeable, functioning as a precision instrument for financial operations.
Color & Tonal Layering: Anchored in "Command Navy" (#0e1629). Use shifting background surface tones (surface_container_low vs surface) instead of 1px solid borders to define boundaries (The "No-Line" Rule).
Typography: Manrope for authoritative financial totals; Inter with tabular lining (tnum) for highly legible 20+ column data grids; Public Sans for technical/audit labels.
Depth & Components: Use "stacking" layers for elevation. Data grids have no vertical borders, relying on row-striping or vertical spacing.
Integrity Visuals: A persistent 4px vertical bar of primary color (Integrity Anchor) denotes data verified against the ledger. Posted transactions receive a surface_dim background and glass overlay denoting immutability.
4. Data Model Overview
The core tables use UUID primary keys and timezone-aware timestamps (TIMESTAMPTZ). All financial posting tables are strictly append-only.

5. Business Rules and Constraints
Zero-Sum Enforcement: Every transaction must mathematically satisfy SUM(journal_entries.amount) = 0. This is enforced via a deferred database trigger that rejects unbalanced transactions at commit.
Immutability: Once a transaction's posted_at is NOT NULL, UPDATE or DELETE commands are hard-blocked on transactions and journal_entries.
Corrections (Contra Pattern): 1. Create a reversing transaction that negates the original lines, referencing original_transaction_id in metadata.

2. Create a new, corrected transaction.
Invoice Posting Atomicity: Posting must occur in a single atomic database transaction:
Validate client GSTIN and place of supply.
Reserve/consume cost layers for stocked items per FIFO/LIFO policy.
Create inventory movement rows and cost layer consumption records.
Create ledger transaction and journal_entries for AR, Revenue, Output GST, Inventory Asset, COGS, and AP.
Mark invoice status = 'posted' and store transaction_id.
Cost Layer Consumption: On inventory issue, consume cost_layers rows in chronological order (FIFO) or reverse (LIFO). Reduce remaining_qty and record the consumed layer IDs on the invoice lines.
Snapshots: At month-end close, compute account_snapshots to accelerate balance queries. Live Balance Formula: snapshot.balance + SUM(journal_entries after snapshot_end).
Tax Handling: Tax components must be separate journal lines with explicit tax_code and gst_type. Supplier and recipient GSTINs are snapshotted on the invoice for B2B transactions.
6. Indexing, Partitioning, and Performance
Primary Indexes:
journal_entries(account_id, created_at) for balance queries.
journal_entries(transaction_id) for transaction retrieval.
invoices(client_id, date) for AR aging reporting.
cost_layers(product_id, received_at) for layer consumption logic.
Partitioning: Partition journal_entries by year (or month for extreme volumes) to maintain query speed on historical data.
Materialized Views: Optional nightly refreshed view for current balances per account to serve executive dashboards.
Batching: Use batched writes for bulk imports, ensuring zero-sum enforcement applies per individual batch transaction.
7. API Design and Event Model
Core REST Endpoints:
POST /api/clients — Create client; automatically writes to client_history.
PUT /api/clients/{id} — Update client; appends to client_history.
GET /api/clients — List clients with filters and pagination.
POST /api/invoices — Create an invoice draft.
POST /api/invoices/{id}/post — Validate, post invoice, create ledger transaction, and trigger e‑invoice adapter.
POST /api/payments — Record payment; create transaction to debit bank and credit AR.
POST /api/inventory/receive — Create receipt, establish cost layer, and post accounting transaction.
POST /api/inventory/issue — Consume cost layers, record movement, and post accounting transaction.
Events and Webhooks (Idempotent):
Emit payloads containing event ID, timestamp, and a data snapshot for: client.created, client.updated, client.activated, invoice.posted (includes IRN), payment.received, and inventory.movement.posted.
8. E-Invoicing Adapter & Payload (India GST)
Responsibilities: Build IRN payload, sign/submit to GSP (GST Suvidha Provider), handle IRP responses, store IRN and signed JSON immutably, and manage retries with idempotency keys.
Minimum Payload Fields: Supplier GSTIN, Recipient GSTIN, Invoice Number, Invoice Date, Invoice Value, HSN Summary, Tax Breakup, Place of Supply, Invoice Lines (with HSN and tax rate).
Storage: Store the IRP-signed response and generated QR code directly on the invoices record.
9. Security, Compliance, and Audit
RBAC: * Admin: Full system access.
Finance: Ledger, AR/AP, Contra-entries.
Sales: Read catalog, draft clients/invoices.
Onboarding: Manage KYC and client activation.
ReadOnly: Auditor access.
Audit Trail: client_history, transactions, and journal_entries are strictly append-only. Access to sensitive fields (GSTIN, PAN, KYC docs) triggers a read-receipt log.
Data Protection: Encrypt KYC storage references and sensitive columns at rest. Enforce TLS 1.2+ in transit.
Tamper Evidence: Implement an optional cryptographic hashing chain for transactions to detect database-level tampering.
Retention: Retain full ledger and audit history indefinitely per statutory financial requirements.
10. Testing, Validation, and Rollout Plan
Testing Strategy:
Unit Tests: Zero-sum enforcement, contra-entry generation, cost layer consumption logic, and state-aware GST calculations.
Integration Tests: End-to-end flow: Create Prospect → Onboard/KYC → Create Draft Invoice → Post → Verify Ledger lines and AR balances; E-invoice flow via GSP sandbox.
Load Tests: Simulate high-volume posting and test snapshot-assisted balance query performance.
Audit Review: Sample audit by an external auditor to validate immutability and traceability prior to go-live.
Rollout Checklist:
Implement core ledger, accounts, transactions, and journal entries.
Add products, cost layers, and inventory movements.
Add invoices and internal posting flows (without external e-invoicing); test end‑to‑end accounting.
Integrate GSP sandbox, validate payloads, and enable production e-invoicing.
Deploy client CRM module and automated onboarding flows.
Run parallel accounting for one month, reconcile against legacy systems, and execute full cutover.
